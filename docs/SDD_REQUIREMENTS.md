# SDD 项目工程要求

本文档定义 `python-git-reproduction` 的 SDD 工程约束。SDD 在本项目中表示“规格、设计、开发、测试同步推进”的工作方式。由于本项目要复现 Git，并且会实现 `reset --hard`、`checkout`、`rm`、引用改写、工作区重写、pack 解包等具有破坏性的能力，因此测试和安全边界必须被视为核心需求，而不是后期补充项。

## 1. SDD 工作原则

每个功能在编码前都必须先明确规格。规格至少要说明输入、输出、会读取或写入哪些 `.pygit` 文件、是否会改写工作区文件、失败时如何回滚。

每个核心模块在实现前都必须有清晰设计。设计至少要说明数据结构、二进制格式、路径边界、锁文件策略、异常处理策略和与官方 Git 的兼容性目标。

每个功能合入前都必须有对应测试。不能只依赖手工试运行，也不能只测试成功路径。

所有 Python 代码必须使用充分中文注释。文件开头必须写意图注释；函数必须写职责、参数、返回值、副作用和异常说明；复杂判断、二进制布局、危险路径操作和兼容性关键点必须写中文关键注释。

## 2. 测试要求

项目必须建立充分的自动化测试。测试目标不是简单提高覆盖率数字，而是证明 `.pygit` 的行为在关键路径上稳定、可恢复、可验证。

必须覆盖以下测试类型：

- 对象库测试：blob、tree、commit、tag 的编码、压缩、解压、SHA-1 校验和损坏对象拒绝。
- Index 测试：Git Index V2 header、entry、padding、flags、stage、checksum、路径排序。
- 引用测试：HEAD、分支、标签、远端引用、符号引用、游离 HEAD、旧值校验、锁文件更新。
- 管道命令测试：`hash-object`、`cat-file`、`write-tree`、`read-tree`、`commit-tree`、`update-ref`。
- 瓷器命令测试：`init`、`add`、`rm`、`status`、`commit`、`log`、`branch`、`checkout`、`switch`、`tag`、`reset`、`stash`。
- 合并测试：最近公共祖先、快进合并、三方合并、冲突标记、index stage 1/2/3、未解决冲突禁止提交。
- Packfile 测试：pack header、idx fan-out table、offset、checksum、对象随机读取、delta 展开。
- 远端测试：`clone`、`fetch`、`push`、非快进拒绝、want/have 协商、pack 传输失败回滚。
- 异常测试：文件缺失、权限错误、对象损坏、index checksum 不匹配、锁文件已存在、写入中断。

凡是会修改磁盘状态的命令，都必须至少包含成功路径测试、失败路径测试和回滚后状态测试。

## 3. 危险指令安全要求

本项目必须特别谨慎处理危险操作。危险操作包括但不限于：

- 删除工作区文件。
- `reset --hard` 强制重写工作区。
- `checkout` 或 `switch` 清理旧版本追踪文件。
- `rm` 删除物理文件。
- `stash pop` 应用并移除 stash 记录。
- `update-ref` 改写分支、标签或远端引用。
- pack 解包时写入大量对象。
- fetch、push、clone 改写远端追踪引用或目标目录。

所有危险操作必须先确认目标路径位于当前仓库工作区内部。任何解析后落在工作区外部的路径都必须拒绝执行。

路径处理必须防御 `..`、绝对路径、符号链接逃逸和平台路径分隔符差异。实现时不能只做字符串前缀判断，必须使用规范化后的真实路径进行边界检查。

删除文件前必须确认该文件是当前 index 或目标 tree 中可解释的追踪文件。默认不能删除未追踪文件，除非命令规格明确要求并且测试覆盖。

工作区重写必须先完成对象完整性校验。不能在确认目标 commit、tree、blob 都可读取之前开始删除旧文件。

## 4. 本机保护要求

`.pygit` 的实现必须假设运行环境是用户真实电脑，而不是一次性容器。任何 bug 都可能误删用户文件，因此默认策略必须保守。

禁止实现任何会递归删除仓库根目录上级目录的逻辑。禁止对用户主目录、桌面、下载目录等非仓库路径执行清理。

涉及目录清理时，必须只清理由 `.pygit/index` 或目标 tree 明确记录的追踪路径。不能简单遍历工作区后删除所有未知文件。

测试中涉及危险操作时，必须使用临时目录。不能在真实项目根目录直接测试破坏性命令。

## 5. 数据一致性要求

写入 index、refs、packfile 或配置文件时必须使用锁文件或临时文件，并通过 `os.replace` 原子替换。

对象写入必须先写临时文件，再原子移动到目标对象路径。对象内容必须先完成 SHA-1 校验再落盘。

如果写入过程中发生异常，正式文件不能变成半写入状态。锁文件残留时，后续命令必须能给出清晰错误，并提供安全恢复策略。

## 6. 官方 Git 交叉验证

关键二进制格式必须尽量使用官方 Git 交叉验证：

- loose object 应能被 `git cat-file` 识别。
- tree 和 commit 对象应能被官方 Git 解析。
- index 文件应尽量兼容官方 Git 的读取规则。
- pack 和 idx 文件应能通过官方 Git 的校验工具。

如果某个阶段暂时不能做到完全兼容，必须在对应文档和测试中标记限制，不能让调用方误以为已经完全兼容。

## 7. 提交质量要求

提交必须保持主题集中。一次 commit 应解决一个清晰问题，不能把无关重构、格式化和功能实现混在一起。

commit 信息必须使用中文，并说明为什么要做这次变更，而不仅仅是描述改了哪些文件。

提交正文应说明设计动机、行为影响、兼容性影响和测试情况。对于危险操作、安全边界、二进制格式和数据一致性相关变更，commit 信息必须更加详细。
